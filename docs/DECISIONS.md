# Technical Decision Log

## D001: 前端框架

| 字段 | 内容 |
|------|------|
| **日期** | 2026-06-07 |
| **上下文** | 需要构建 WebSocket 实时行情前端 |

## D003: 后端框架

| 字段 | 内容 |
|------|------|
| **日期** | 2026-06-07 |
| **上下文** | 需要提供 REST API + WebSocket 服务 |
| **选项** | FastAPI / Node.js Express+ws / Tornado |
| **决策** | **FastAPI** |
| **理由** | 原生 WebSocket 支持，async/await 异步，与现有 Python 数据栈无缝集成，自动生成 OpenAPI 文档 |

## D004: 数据存储

| 字段 | 内容 |
|------|------|
| **日期** | 2026-06-07 |
| **上下文** | 分钟K线时序数据持久化 |
| **选项** | SQLite / TimescaleDB / InfluxDB / Parquet 文件 |
| **决策** | **SQLite(开发) → TimescaleDB(生产)** |
| **理由** | SQLite零依赖适合开发验证；TimescaleDB 自动分区/压缩/连续聚合适合生产（用户选择升级到TSDB） |

## D005: 实时更新策略

| 字段 | 内容 |
|------|------|
| **日期** | 2026-06-07 |
| **上下文** | 前端需要实时显示最新行情 |
| **选项** | 1分钟轮询+WS推送 / WebSocket长连接+tick推送 / SSE |
| **决策** | **1分钟轮询 + WebSocket推送** |
| **理由** | 数据源本身是分钟级，无需tick级推送。后端一分钟拉取一次最新K线，通过WS推送到前端 |

## D009: 数据推送频率

| 字段 | 内容 |
|------|------|
| **日期** | 2026-06-07 |
| **上下文** | 需要平衡实时性和频控 |
